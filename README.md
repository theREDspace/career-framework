# Engineering Career Framework

This repository contains the Individual Contributor (IC) engineering career framework used at REDspace. It defines the expectations, behaviours, and competencies for each level of the software development track — from a developer learning the fundamentals to a Principal setting company-wide technical strategy.

We're sharing it publicly in the spirit of transparency and in hopes it's useful to other teams building or refining their own frameworks.

---

## Levels

The IC track is organized into three tiers across seven levels:

| Code | Title | Theme |
|---|---|---|
| SD1 | Software Developer | Adopts |
| SD2 | Software Developer | Competence |
| SD3 | Software Developer | Trust |
| SSD1 | Senior Software Developer | Leads |
| SSD2 | Senior Software Developer | Grows |
| SSD3 | Senior Software Developer | Architects |
| Principal | Principal Software Developer | Strategizes |

> **SSD1 and SSD2 are recognized as terminal levels.** Engineers who operate at a high bar within these levels are not expected to pursue further promotion to remain at the company.

---

## Competency Areas

Every level is assessed across six equally-weighted competency areas:

- **Technology** — Technical knowledge and how it's applied
- **Solution** — Problem-solving, design, and delivery
- **People** — Collaboration, mentorship, and communication
- **Process** — How work is planned, managed, and improved
- **Influence** — Impact on team, department, and company culture
- **Client Satisfaction / Business Value** — Understanding why the work matters and de-risking client relationships through technical excellence

No single area outweighs another. Strength in one area doesn't compensate for a consistent gap in another.

---

## How to Use This Framework

The ladder is a **conversation tool, not a checklist**. The bullet points under each competency are illustrative examples of behaviours — not an exhaustive list of requirements. Promotion is based on consistently demonstrating the core focus of the next level across all six areas, typically over a period of at least six months.

See [`faq.md`](./software-developer/faq.md) for answers to common questions about promotions, terminal levels, specialization, and how the ladder is maintained.
